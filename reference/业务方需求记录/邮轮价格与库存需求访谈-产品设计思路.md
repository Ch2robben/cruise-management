# 邮轮价格与库存产品设计思路

> 基于访谈纪要：`邮轮价格与库存需求访谈-meeting-notes.md`
> 目标：把访谈中的业务口径转成可落地的产品设计框架，支撑后续原型、PRD 和研发拆解。

## 1. 设计判断

这次访谈的核心不是“做一个价格表”，而是要设计一套可配置的邮轮价格与库存规则系统。

邮轮业务的复杂点在于：价格不是单一字段，库存也不是单纯房间数。价格会同时受到航线、航段、房型、入住序位、游客类型、经销商政策、区域政策、临时促销、库存名额、占床规则和结算口径影响。

因此产品设计上不能把它做成“计调逐项录价格”的后台表单，而应抽象成：

1. 基础价格层：解决不同航线、航段、房型的标准价格。
2. 规则政策层：解决谁能享受什么价格、什么条件下生效、优先级如何。
3. 库存约束层：解决某个价格政策有多少名额、用完后怎么回退。
4. 游客计算层：解决一个订单里每个人按什么身份、序位、占床规则计价。
5. 财务结算层：解决订单最终价、经销商充值、经营人员提成和后续对账。

## 2. 产品设计目标

### 2.1 对业务方

- 能配置多航线、多航段、多房型的价格。
- 能配置区域、口岸、经销商集合、临时政策等价格政策。
- 能支持临时促销和库存名额控制。
- 能处理儿童、婴儿、加床、不占床、高级房整间售卖等复杂价格规则。
- 能保留基准价、结算价、优惠金额，用于财务和提成。

### 2.2 对计调人员

- 不需要逐个经销商、逐条航线重复录价格。
- 常规价格通过批量投放完成。
- 临时政策可快速加一层规则，不破坏常规价格。
- 有明确的校验提示，避免未配置系数导致错误预订。

### 2.3 对财务人员

- 能看到充值来源、支付状态、手动提单、水单匹配状态。
- 财务审核后能先增加额度，但后续仍能追踪水单是否补齐。
- 能通过提醒和结单机制关闭长期未匹配事项。

### 2.4 对研发实现

- 价格计算逻辑应规则化、可解释、可追溯。
- 每次订单计价都应保存计算快照，避免政策变更后历史订单不可还原。
- 规则配置应优先做成组合式模型，而不是硬编码每种业务场景。

## 3. 核心产品模型

### 3.1 基础对象

| 对象 | 设计含义 | 关键字段 |
| --- | --- | --- |
| 船舶 | 承载航次、房型、床位与核载信息 | 船名、核载上限、可用状态 |
| 航线 | 一条完整运营线路 | 起点、终点、途经港口、长短线标识 |
| 航段 | 航线中的可售区间 | 上船港、下船港、里程/天数、所属航线 |
| 航次 | 某条航线在某日期的具体发班 | 船舶、航线、出发日期、可售状态 |
| 房型 | 船上的售卖单元类型 | 标间、家庭房、高级房、可住人数、床位数 |
| 入住类型 | 游客身份类型 | 成人、儿童、婴儿、外宾等 |
| 经销商 | 外部销售主体 | 所属区域、所属口岸、维护经营人员 |
| 经营人员 | 内部销售/维护人员 | 归属部门、维护经销商、提成归属 |

### 3.2 价格对象

| 对象 | 设计含义 |
| --- | --- |
| 基准价 | 某航段、房型、时间范围下的基础价格 |
| 月度价格 | 基准价的一种维护周期，通常按月投放 |
| 价格政策 | 对基础价格的调整规则，例如区域优惠、口岸价、临时促销 |
| 价格公式 | 支持 `基准价 * 系数 + 固定值`，兼容纯系数和固定价 |
| 政策库存 | 某个价格政策可使用的名额或库存 |
| 政策优先级 | 多政策命中时的排序和回退机制 |
| 订单计价快照 | 下单时实际使用的价格、政策、公式和计算结果 |

## 4. 价格体系设计

### 4.1 分层价格结构

建议把价格拆成四层：

| 层级 | 作用 | 示例 |
| --- | --- | --- |
| 第1层：基础价 | 定义某航段、房型、日期的正常价 | 重庆-宜昌，标间，成人基础价5000 |
| 第2层：入住价 | 根据第几人、成人/儿童/婴儿、占床情况计算 | 第3人儿童不占床 = 成人价 * 0.7 |
| 第3层：渠道政策价 | 根据经销商、区域、口岸等规则调整 | 上海区域减300 |
| 第4层：临时政策价 | 针对销售不佳航次的临时促销 | 某航次前10名减200 |

这样做的好处是：基础价和政策价解耦，月度调价时主要动基础价；政策相对稳定；临时促销可叠加但可控。

### 4.2 基准价不是单个全局价格

访谈中明确提到，不同航段都要有自己的基准价。因此不能设计成“产品一个基准价”，而应设计成：

```text
基准价 = 航次 / 航线 / 航段 / 房型 / 日期范围 / 入住价格组
```

如果后续支持多航段销售，必须允许同一航次下，不同上下船区间使用不同价格。

### 4.3 价格政策配置方式

价格政策建议配置为规则对象：

| 字段 | 说明 |
| --- | --- |
| 政策名称 | 例如“上海区域优惠”“重庆口岸价”“5月25日临时促销” |
| 政策类型 | 区域政策、口岸政策、经销商集合政策、临时促销、团队优惠 |
| 适用航线/航段 | 可指定全部、部分航线或具体航段 |
| 适用航次 | 临时促销可指定具体航次 |
| 适用客源 | 身份证地区、外宾国家、经销商集合等 |
| 价格算法 | 直减、固定价、系数、阶梯价、`m+n`公式 |
| 政策库存 | 可为空；为空表示不限名额 |
| 优先级 | 多政策命中时使用 |
| 生效时间 | 起止日期 |
| 审批状态 | 草稿、审批中、已生效、已失效 |

### 4.4 政策命中流程

建议计价时按以下流程：

```text
选择航次/航段/房型
  -> 读取基础价
  -> 根据游客信息计算入住序位价
  -> 根据经销商/客源/区域匹配价格政策
  -> 按优先级排序
  -> 判断政策库存是否可用
  -> 使用最高优先级可用政策
  -> 如果库存不足，回退到下一优先级政策
  -> 生成订单价格明细和计价快照
```

## 5. 库存体系设计

库存不能只看房间数量，需要拆成三类：

| 库存类型 | 含义 | 设计重点 |
| --- | --- | --- |
| 房间/床位库存 | 实际可售房间和床位 | 控制占床销售 |
| 核载人数 | 船舶安全承载人数 | 控制不占床儿童/婴儿后的总人数 |
| 政策库存 | 某个优惠政策可用名额 | 控制优惠使用量 |

### 5.1 房间库存

房间库存用于控制是否还有可售房间、可售床位。标间可能支持拼房，高级房更接近整间售卖。

### 5.2 核载人数

核载人数通常大于床位数。不占床儿童/婴儿可能不消耗床位，但仍应计入船舶核载人数。

### 5.3 政策库存

政策库存是价格政策的一部分。比如某临时促销只有10个名额，使用完后不应阻止预订，而是应回退到下一个可用价格政策。

## 6. 房型与入住计价设计

### 6.1 不要按“票种”设计

不能简单设计成人票、儿童票、婴儿票。更准确的模型是：

```text
房型 + 第几位入住人 + 入住类型 + 是否占床/加床/不占床 = 价格规则
```

例如：

| 房型 | 第几人 | 入住类型 | 占床规则 | 价格规则 |
| --- | --- | --- | --- | --- |
| 标间 | 第1人 | 成人 | 占床 | 成人基准价 |
| 标间 | 第2人 | 成人 | 占床 | 成人基准价 |
| 标间 | 第3人 | 儿童 | 加床/不占床 | 成人价 * 0.7 |
| 家庭房 | 第5人 | 儿童 | 不占床 | 成人价 * m + n |
| 高级房 | 单人入住 | 成人 | 整间 | 按房间价或双床位合并价 |

### 6.2 入住优先级

当一个房间有多名游客时，需要先排序再计价：

```text
成人优先占床 > 儿童占床/加床 > 婴儿
```

典型场景：两大一小入住标间，应是两个成人占床，儿童加床或不占床，不能反过来让儿童占床、成人加床。

### 6.3 未配置即禁止预订

如果某个房型、入住序位、入住类型没有配置价格公式，应禁止预订，而不是默认给0或默认成人价。否则会产生错误订单。

## 7. 订单价格明细设计

订单需要保存两类价格：

| 类型 | 作用 |
| --- | --- |
| 计算过程价格 | 用于解释为什么这个游客是这个价格 |
| 结算结果价格 | 用于收款、对账、提成、报表 |

建议订单明细保存：

- 航次、航段、房型。
- 游客入住类型。
- 第几人入住。
- 是否占床/加床/不占床。
- 基准价。
- 命中的价格政策。
- 政策优惠金额。
- 实际结算价。
- 经营人员归属。
- 计价快照版本。

这样后续即使价格政策变了，也能还原当时订单为什么这么算。

## 8. 充值与财务流程设计

### 8.1 两种充值路径

| 路径 | 流程 |
| --- | --- |
| 线上充值 | 经销商支付 -> 通联返回成功 -> 增加可用额度 -> 记录支付流水 |
| 手动提单 | 经销商/经营人员提交转账信息 -> 财务审核 -> 增加额度 -> 后补/匹配水单 |

### 8.2 手动充值状态机

建议状态如下：

```text
待提交
  -> 待财务审核
  -> 审核通过，额度已增加
  -> 待水单匹配
  -> 已匹配，已结单

异常分支：
待财务审核 -> 审核驳回
待水单匹配 -> 逾期未匹配 -> 已提醒 -> 人工结单
```

### 8.3 风控提醒

水单不是额度增加的强前置条件，但必须有追踪机制：

- 超过N天未匹配水单，提醒经销商、经营人员、财务。
- 提醒后仍未处理，进入人工结单。
- 结单必须填写原因。
- 后续可按经销商、经营人员、财务人员维度查看未匹配记录。

## 9. 组合产品设计

组合产品先不要一开始就做复杂分账，建议分两期：

### 一期：统一收款，线下结算

- 订单中允许挂接搭售产品。
- 用户统一支付。
- 财务后台能看到主产品和搭售产品的拆分金额。
- 线下定期结算。

### 二期：线上分账

- 依赖通联支付分账能力确认。
- 支持不同主体结算。
- 支持结算规则、分账比例、退款联动。

## 10. 后台原型建议

### 10.1 第一优先级页面

| 页面 | 目的 |
| --- | --- |
| 航段基准价配置 | 配置不同航段、房型、日期的基础价格 |
| 价格政策配置 | 配置区域、口岸、经销商集合、临时促销 |
| 政策优先级与库存 | 配置政策排序、名额、用完回退 |
| 房型入住规则 | 配置第几人、成人/儿童/婴儿、占床/加床价格公式 |
| 订单试算器 | 输入航次、房型、游客、经销商，展示计价过程 |

### 10.2 第二优先级页面

| 页面 | 目的 |
| --- | --- |
| 经销商集合管理 | 管理区域、口岸、经销商归属 |
| 手动充值审核 | 财务审核线下充值 |
| 水单匹配与提醒 | 处理未匹配流水 |
| 组合产品配置 | 配置搭售产品和结算方式 |
| 提成报表 | 按经营人员查看价差和提成数据 |

## 11. 订单试算器是关键原型

建议一定做一个“订单试算器”原型，因为这套规则复杂，光看配置页业务方很难确认理解是否一致。

试算器输入：

- 航线/航段。
- 航次日期。
- 房型。
- 经销商。
- 游客列表：成人/儿童/婴儿、身份证地区/外宾国家。
- 是否拼房、是否加床、是否占床。

试算器输出：

- 命中的基准价。
- 第几人入住规则。
- 命中的价格政策。
- 政策优先级。
- 政策库存消耗。
- 每位游客价格。
- 订单总价。
- 经营人员归属。

这个页面可以作为和甲方对齐规则的核心工具。

## 12. 推荐的产品落地顺序

### 第一步：先跑通常规价格

- 航线/航段。
- 航次。
- 房型。
- 月度基准价。
- 成人标准入住价格。

### 第二步：加入入住序位规则

- 第1人、第2人、第3人、第4人。
- 成人、儿童、婴儿。
- 占床、加床、不占床。
- 未配置禁止预订。

### 第三步：加入经销商价格政策

- 区域集合。
- 口岸集合。
- 经销商集合。
- 政策优先级。
- 政策库存。

### 第四步：加入临时政策和审批

- 临时降价。
- 返利点。
- 政策审批。
- 政策库存用完回退。

### 第五步：加入财务与提成

- 充值额度。
- 手动提单。
- 水单匹配。
- 经营人员归属。
- 价差提成报表。

### 第六步：加入组合产品

- 搭售产品。
- 统一收款。
- 线下结算。
- 后续再评估线上分账。

## 13. 需要避免的设计误区

1. 不要把经销商价做成每个经销商一张价目表，维护和审批都会失控。
2. 不要只设计成人票、儿童票、婴儿票，实际规则按入住序位和占床情况计算。
3. 不要把库存只理解为房间库存，还要有床位、核载、政策名额。
4. 不要让折扣作为唯一价格算法，整数销售和固定值场景很多。
5. 不要只保存最终价，必须保存基准价、政策、优惠、计价快照。
6. 不要把水单匹配做成充值成功的唯一前置条件，业务上可能先审核加额度。
7. 不要一开始强依赖线上分账，组合产品可先统一收款、线下结算。

## 14. 一句话设计原则

邮轮价格系统的核心不是录价格，而是用“航段基准价 + 入住序位公式 + 客源政策规则 + 政策库存回退 + 订单计价快照”把复杂业务变成可配置、可解释、可追溯的计价引擎。
