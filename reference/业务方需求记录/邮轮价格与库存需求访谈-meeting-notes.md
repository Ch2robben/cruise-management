## 会议背景

本次会议围绕邮轮业务中最核心、最复杂的部分展开：价格、仓位/库存、航线航段、经销商政策、入住人数与占床规则。甲方强调，邮轮整体设计中，除预订、供应商入驻、权限等常规能力外，价格和库存是成败关键，需要在本周内产出原型设计并持续核对成果。

## 核心结论

1. 邮轮价格不应简单按“供应商单独价格”维护，而应通过“基础价格 + 价格政策 + 适用范围 + 优先级 + 库存名额”的方式配置。
2. 月度正常价/基准价会变化，但渠道、口岸、区域、代理集合等价格政策关系相对稳定。
3. 价格政策需要支持区域、口岸、经销商集合、临时促销等多类适用范围，并根据客人信息进行命中判断。
4. 价格政策和库存强相关，同一个政策可绑定有限名额，例如20个库存中10个减200、10个减100。
5. 临时政策用完后，需要按优先级回退到上一级政策，例如上海政策用完后回退区域政策，再回退口岸政策。
6. 邮轮价格计算不能只按成人、儿童、婴儿划分，还需要按房间内第1人、第2人、第3人、第4人等入住序位配置价格系数。
7. 不同航线/航段需要独立设置基准价，航段不同，价格系数关系可能不同。
8. 外币充值、手动充值、财务流水、水单匹配、逾期提醒和结单机制都需要纳入风控设计。
9. 组合产品目前优先考虑统一收款、线下结算；线上分账待确认通联支付能力。
10. 经营人员本质上也可视为内部经销商，订单必须挂接经营人员，用于后续提成计算。

## 需求要点

### 1. 价格模型

| 要点 | 说明 | 依据 |
| --- | --- | --- |
| 基准价/月度正常价 | 每个月会出各房型正常价，可视为系统价格计算的基础价格。 | 04:22-06:17 |
| 价格政策 | 价格政策可按口岸、区域、经销商集合、临时政策配置。 | 06:17-13:56 |
| 政策稳定性 | 价格政策相对稳定，月度价格更常变化。 | 06:17-07:13 |
| 整数销售 | 甲方倾向整数销售，折扣导致小数时价值不大。 | 08:08-08:23 |
| 政策优先级 | 需要支持多政策命中后的优先级排序和回退。 | 09:04-13:56 |
| 政策库存 | 价格政策需绑定库存/名额。 | 09:22-13:56 |
| 临时促销 | 航次销售不好时，可临时降价或提高返利点。 | 11:33-12:34 |

### 2. 经销商与适用范围

价格政策不应逐个经销商维护，否则计调和审批成本过高。更合理的方式是维护“集合”，例如区域集合、口岸集合、经销商集合，再由集合包含具体经销商。审批时审核政策本身，而不是逐个经销商逐条航线审核价格。

适用范围判断需要基于客人信息：

| 客人类型 | 判断方式 |
| --- | --- |
| 国内游客 | 可根据身份证判断到地级市，必要时到县级 |
| 外宾 | 可按国家维度判断，或手动选择 |
| 经销商/代理 | 根据所属集合、区域、口岸、临时政策判断 |

### 3. 库存与价格联动

价格政策需要与库存名额绑定。例如同一个代理商可获得20个库存，其中10个库存减200，另外10个库存减100。系统需要支持：

- 政策绑定库存数量。
- 政策库存消耗。
- 政策用完后的优先级回退。
- 同一航次、同一区域、同一客户信息下的多政策命中排序。
- 临时政策库存和常规政策库存并存。

### 4. 优惠与团队政策

外宾团队存在“16免1”政策，即16个人按15个人计算。但免掉的人不应简单从总人数里扣除，而应落到价格最低的人身上，便于订单和结算记录准确。

需要支持的优惠逻辑包括：

| 类型 | 说明 |
| --- | --- |
| 16免1 | 面向团队，满足人数后按最低价人员免除 |
| 阶梯价 | 不同名额对应不同优惠，例如前10个名额减200 |
| 返利点 | 可用于刺激销售较差航次 |
| 临时改价 | 特殊情况走改价/审批，不作为常规政策 |

### 5. 充值与财务风控

经销商或经营人员采用“先充值后消费”。充值分两类：

| 充值方式 | 说明 |
| --- | --- |
| 线上充值 | 通过通联支付，支付成功后增加可用额度 |
| 手动提单 | 经销商或经营人员通过线下转账、对公转账后发起提单 |

风控和财务流程要求：

- 通联支付成功后，先按支付成功状态增加额度。
- 外币支付暂不深入处理汇率差，先视为通联或支付侧完成换算。
- 手动提单可以由经销商发起，也可以由维护经销商的经营人员发起。
- 水单与充值单匹配不是额度增加的强前置条件，财务审核通过即可增加额度。
- 即使财务先审核通过，后续仍需补齐流水/水单匹配。
- 系统需要设置逾期提醒机制，同时提醒经销商、经营人员和财务。
- 如果长期未补齐，需要有结单机制，并记录未匹配原因。

### 6. 组合产品

组合产品主要是长线游搭配短线游，例如长江游轮搭售武汉两天游、重庆两天游，未来也可能搭南京景点。

当前结算建议：

| 方案 | 状态 | 说明 |
| --- | --- | --- |
| 统一收款，线下结算 | 优先考虑 | 例如统一收到重庆公司邮轮账户，再由财务定期线下结算 |
| 线上分账 | 待确认 | 取决于通联支付是否支持 |

价格侧至少需要区分售价、结算价、成本价等价格字段，但这些字段不一定全部存在固定系数关系，门市价/建议零售价可能长期不变。

### 7. 订单归属与提成

每个订单都需要挂接经营人员。经营人员可理解为内部经销商或内部对外维护人，其作用包括：

- 维护经销商关系。
- 协助处理线下转账和充值。
- 作为订单归属人。
- 支撑提成计算。

系统需要记录：

| 价格字段 | 用途 |
| --- | --- |
| 系统常规投放价/基准价 | 作为价差和提成计算基础 |
| 实际结算价 | 经优惠、折扣、溢价后的最终价格 |
| 减免金额/价差 | 用于提成档位判断 |
| 每位游客价格明细 | 成人标间、加床、不占床等逐人记录 |

提成逻辑可能基于基准价与实际结算价之间的价差计算。历史上不同价差对应不同提成，当前是否仍按此规则需要再与人资确认。

### 8. 航线、航段与基准价

如果只卖重庆到宜昌，现有逻辑可相对简单；若支持多航段，则每个航段都需要设置基准价，例如：

| 航段 | 价格要求 |
| --- | --- |
| 重庆-宜昌 | 独立基准价 |
| 重庆-武汉 | 独立基准价 |
| 其他航段 | 独立基准价及可能不同的系数关系 |

基准价会随航线/航段不同而不同，系数关系全年可能相对稳定，但也需要可配置。

### 9. 房型、入住序位与占床规则

甲方强调，价格不能只按成人、儿童、婴儿配置，而要按“房型 + 第几人入住 + 入住类型 + 是否占床/加床”综合判断。

基础规则：

| 规则 | 说明 |
| --- | --- |
| 标准间 | 通常最多4人，前两人占床 |
| 家庭房 | 可能5人、3张床 |
| 高级房 | 更接近按房间售卖，不与陌生人拼房 |
| 拼房 | 主要存在于标间，两个陌生人可分别占床，价格一致 |
| 儿童 | 不能单独入住，也不能单独预订 |
| 婴儿 | 带婴儿必须单独入住，不能拼房 |
| 占床优先级 | 成人 > 儿童 > 婴儿 |

价格计算需要考虑：

- 第1人、第2人、第3人、第4人等入住序位。
- 成人、儿童、婴儿等入住类型。
- 是否占床、加床、不占床。
- 不同房型可住人数上限。
- 系数关系，例如儿童按成人价0.7。
- 固定值与系数组合，例如 `价格 = 成人价 * m + n`，其中 `m` 为系数，`n` 为固定金额。
- 未配置系数或公式时，应禁止预订。

特殊场景：

| 场景 | 处理 |
| --- | --- |
| 婴儿占床 | 可临时投放，按儿童0.7等规则计算；不投放则不可预订 |
| 两大一小 | 两个成人优先占床，儿童加床或不占床 |
| 高级房单人入住 | 单人占用整间房，可能把两个床位系数合并到一个人身上 |
| 高级房一大一婴儿 | 可按1+1理解，但走高级房自己的系数关系 |

### 10. 核载、床位与库存

船舶存在两个概念：

| 概念 | 说明 |
| --- | --- |
| 床位数 | 实际可占床数量，是库存/仓位控制的核心 |
| 核载上限 | 安全载客上限，通常大于床位数 |

库存主要受床位数限制。例如床位数600，但核载上限可能接近1000。儿童或婴儿不占床时，人数可能超过床位数，但不能超过核载上限。系统需要同时记录占床与核载影响。

### 11. 航线相关费用

以下费用或配置与航线相关：

| 类型 | 说明 |
| --- | --- |
| 小费 | 多数针对外宾，可能提前收取 |
| 劳务费 | 长线和短线不同 |
| 楼层费 | 长江一号、长江二号使用；长江三号按楼层房间卖 |
| 楼层费航段化 | 多航段时，楼层费可能也需要按航段配置 |

楼层费不是简单按里程区间配置，不建议直接按0-500公里、500公里以上等方式建模。甲方提到更接近价格和内部机构相关逻辑。

## 设计约束

1. 不要把价格政策设计成逐个经销商逐条维护，必须抽象成集合和规则。
2. 价格政策必须支持优先级、库存名额、临时政策和回退逻辑。
3. 房型价格计算必须支持穷举式规则，不应仅用成人/儿童/婴儿三个票种简化。
4. 未配置价格系数或公式的入住组合，应禁止预订，避免异常订单。
5. 手动充值不能只依赖流水匹配，必须兼容财务先审核、后补水单的实际流程。
6. 提成计算需要保留基准价、实际结算价和价差，不应只存最终订单金额。
7. 组合产品暂不强依赖线上分账，需先支持统一收款和线下结算。
